## 任务三：编写流水线配置

### 流水线文件位置
`.github/workflows/ci.yml`

### 各阶段说明

流水线分为3个Job（阶段），通过 `needs` 依赖实现质量门禁：

**Job 1：🧪 代码检查与测试**

| 步骤 | 名称 | 说明 |
|------|------|------|
| 1 | 📥 拉取代码 | `actions/checkout@v4` 自动克隆仓库 |
| 2 | 🔧 配置Node.js | 设置 Node.js 20 运行环境 |
| 3 | 🔍 静态检查-HTML语法验证 | shell脚本检查所有HTML文件语法和结构 |
| 4 | 📋 文件完整性预检 | shell脚本检查6个核心HTML文件 |
| 5 | 🧪 运行自动化测试（质量门禁） | `npm test`，失败则整个Job失败 |
| 6 | 📊 生成测试覆盖率报告 | 始终执行，统计文件覆盖率 |
| 7 | 📁 归档测试报告 | 上传到 Actions Artifacts（30天） |

**Job 2：📦 构建与打包**（依赖 Job 1 通过）

| 步骤 | 名称 | 说明 |
|------|------|------|
| 8 | 📥 拉取代码 | 克隆仓库 |
| 9 | 🔧 配置Node.js | 设置运行环境 |
| 10 | 📦 打包构建产物 | 复制所有前端文件到 dist/ |
| 11 | 📁 归档构建产物 | 上传 dist/ 到 Artifacts（7天） |

**Job 3：🔔 质量门禁通知**（始终执行）

| 步骤 | 名称 | 说明 |
|------|------|------|
| 12 | 📊 输出质量门禁结果 | 汇总 test 和 build 状态，输出报告 |
| 13 | 🚫 测试失败时阻止合并 | 测试失败时输出拦截信息 |

### 故意制造失败

**方法：** 将 `package.json` 中的 `test` 命令改为运行 `run-tests-broken.js`：
```json
"test": "node tests/run-tests-broken.js"
```

**失败日志（预期）：**
```
❌ FAIL: [故意失败] 700分排名是第1名（断言错误）
   原因: 期望 "1"，实际得到 "2000"
❌ FAIL: [故意失败] 检查不存在的文件 backend.jar
   原因: 期望 true，实际得到 false

💥 存在失败的测试！CI流水线将标记此次构建为 FAILED
```

**失败原因分析：**
- 断言错误：700分预测排名是2000，但测试错误地期望排名为1
- 文件缺失：backend.jar 是Java后端文件，本项目暂未集成

**修复方法：** 将 `package.json` 改回 `node tests/run-tests.js`，重新推送，流水线恢复绿色

## 任务五：质量门禁与通知

### 质量门禁配置说明

本项目采用 **多Job依赖** 实现质量门禁，流水线分为3个阶段：

```
🧪 代码检查与测试  ──(通过)──→  📦 构建与打包  ──(始终)──→  🔔 质量门禁通知
        │                            │                          │
   测试失败则                       依赖 test 通过               汇总结果
   后续全部跳过                     test 失败则跳过              并输出报告
```

**（1）配置测试失败即中止后续阶段**

```yaml
build:
  needs: test          # ← 质量门禁核心：test Job 必须通过
  if: success()        # ← 双重保障

notify:
  needs: [test, build]
  if: always()         # ← 通知始终执行，汇总门禁结果
```

当测试失败时：
- `📦 构建与打包` Job 被跳过（不会打包有问题的代码）
- `🔔 质量门禁通知` Job 仍然运行，输出 ❌ 门禁拦截信息

**（2）可选配置：静态检查、覆盖率、构建产物归档**

| 配置项 | 说明 | 实现方式 |
|--------|------|----------|
| 🔍 HTML静态检查 | 检查所有HTML文件语法和结构完整性 | shell脚本逐文件验证 |
| 📊 测试覆盖率 | 统计被测试覆盖的文件比例 | 自动计算文件覆盖率 |
| 📁 测试报告归档 | 保存每次测试执行日志 | `upload-artifact` 保留30天 |
| 📁 构建产物归档 | 保存打包好的前端文件 | `upload-artifact` 保留7天 |
| 🔔 门禁通知 | 汇总各阶段状态，测试失败时输出阻止信息 | notify Job 输出报告 |

**（3）总结 CI/CD 对项目质量和协作效率的影响**

**对项目质量的影响：**
1. **自动验证**：每次提交代码都自动跑18个测试，有人改了算法弄错了能立刻发现
2. **质量门禁**：测试没过就不让构建，不会把有问题的版本发布出去
3. **静态检查**：在测试之前先检查HTML语法，提前发现低级错误
4. **可追溯**：每次构建都有报告和日志，能查到是哪次提交引入了bug

**对协作效率的影响：**
1. **减少人工检查**：以前要手动检查文件有没有少，现在流水线自动检查
2. **快速反馈**：推送代码后几分钟内就能知道有没有问题，不用等队友
3. **信心提升**：绿色的流水线 = 代码可以放心合并，不需要反复确认
4. **分工明确**：测试、构建、通知三个Job各司其职，日志更容易定位问题
