### 故意制造失败

**方法：** 将 `package.json` 中的 `test` 命令改为运行 `run-tests-broken.js`：
```json
"test": "node tests/run-tests-broken.js"
```

**失败日志（预期）：**
```
❌ FAIL: [故意失败] 700分排名是第1名（断言错误）
   原因: 期望 "1"，实际得到 "2000"
❌ FAIL: [故意失败] 检查不存在的文件 backend.jar
   原因: 期望 true，实际得到 false

💥 存在失败的测试！CI流水线将标记此次构建为 FAILED
```

**失败原因分析：**
- 断言错误：700分预测排名是2000，但测试错误地期望排名为1
- 文件缺失：backend.jar 是Java后端文件，本项目暂未集成

**修复方法：** 将 `package.json` 改回 `node tests/run-tests.js`，重新推送，流水线恢复绿色

## 任务五：质量门禁与通知

### 质量门禁配置说明

本项目配置了以下质量门禁：

**1. 测试失败即中止**
```yaml
- name: 🧪 运行自动化测试
  run: npm test
  # 测试脚本 exit(1) → 后续步骤自动跳过
```

**2. 构建打包仅在测试通过后执行**
```yaml
- name: 📦 打包构建产物
  if: success()   # 只有测试通过才执行
```

**3. 测试报告始终生成（便于排查）**
```yaml
- name: 📊 生成测试报告
  if: always()    # 无论成功失败都生成
```

### 构建产物说明
| 产物名称 | 内容 | 保留时间 |
|----------|------|----------|
| `test-report-{编号}` | 测试执行日志 | 30天 |
| `dist-{编号}` | 打包好的前端文件 | 7天 |

### CI/CD 对项目质量和协作效率的影响总结

**对项目质量的影响：**
1. **自动验证**：每次提交代码都自动跑18个测试，有人改了算法弄错了能立刻发现
2. **质量门禁**：测试没过就不让打包，不会把有问题的版本发布出去
3. **可追溯**：每次构建都有报告和日志，能查到是哪次提交引入了bug

**对协作效率的影响：**
1. **减少人工检查**：以前要手动检查文件有没有少，现在流水线自动检查
2. **快速反馈**：推送代码后几分钟内就能知道有没有问题，不用等队友
3. **信心提升**：绿色的流水线 = 代码可以放心合并，不需要反复确认
